# Phase 21 — Per-part pages `/part/[kind]/[slug]`

> Detailed brief for the autonomous loop. Generated by
> `/ship-a-phase` 2026-05-10 from the PHASE_CANDIDATES expand
> pass-2 row (score 7.5) + the build plan one-liner. Promoted via
> `/oversight` 2026-05-10.

## Outcome

Every record in `data/switches/`, `data/keycap-sets/`, and
`data/boards/` is reachable at a canonical URL of the form
`/part/<kind>/<slug>`, where `<kind>` is one of `switch`,
`keycap-set`, `board`. Each page renders the part's identity,
its kind-specific spec sheet, the part description, and a
"mentioned in" rail of every article whose
`frontmatter.mentionedParts` references this part. The
`MentionedPartsRail` items on every article become anchors to
those pages; the pass-6 #15 regression-guard test relaxes
naturally. Sitemap enumerates every part URL.

## Why

Two independent signals point at the same shape:

1. **Critique pass-6 #15** ("Build sheet cards style as
   clickable but render no anchors") shipped path (b) — strip
   the card chrome — as an interim affordance fix at `7c9a128`.
   The row's path (a) named "wait for phase 4 to ship per-part
   pages at `/part/[slug]`" — but phase 4 was URL contract /
   e2e infra, not per-part pages. Path (a) is what this phase
   delivers.

2. **Phase 20 backfill** populated 13+ entity records across
   switches / keycap-sets / boards, all of which currently
   live as JSON with no detail surface. Sitemap walks none of
   them.

As the article catalog grows, the navigation hole grows with
it. Every article that uses `mentionedParts` (currently 7 of
12) names parts the reader can't click through to.

## Routes

```
apps/web/src/app/part/
├── [kind]/
│   ├── [slug]/
│   │   ├── page.tsx                    # the canonical part page
│   │   ├── loading.tsx                 # Suspense skeleton
│   │   ├── not-found.tsx               # 404 for unknown slug
│   │   └── opengraph-image.tsx         # per-part OG image (deferred — phase 23+)
│   └── page.tsx                        # kind-index page (e.g. /part/switch lists every switch)
```

`<kind>` is a literal segment (one of `switch`, `keycap-set`,
`board`) — not a discovery step. The page resolves the slug
against the kind's loader (`getSwitchBySlug` /
`getKeycapSetBySlug` / `getBoardBySlug`) and 404s if absent.

The kind-index page (`/part/<kind>`) is a thin index — list of
every part of that kind, ordered alphabetically (or by status:
in-production > sold-out / discontinued at the bottom). Helps
SEO + provides a faceted browse path. Optional but cheap.

OG images for individual parts are **deferred** — placeholder
defaults inherit from the root opengraph-image.tsx. A future
phase or `/ship-asset` drain pass adds per-part SVG art.

## Components

```
apps/web/src/components/part/

├── PartIndexCard.tsx          # row in the kind-index page

    ├── PartSpec.test.tsx
    ├── PartBody.test.tsx
    ├── MentionedInArticles.test.tsx
    └── PartIndexCard.test.tsx
```

### `<PartHero>`

H1 is the part name. Below: a kind eyebrow ("Switch" /
"Keycap set" / "Board") in mono uppercase tracking, plus the
vendor name as a `<Link>` to the vendor's vendor page (if it
exists; otherwise plain text). Status pill on the right
(in-production / discontinued / in-stock / sold-out). Mirrors
`<ArticleHero>`'s spacing rhythm.

### `<PartSpec>`

Kind-specific spec rows rendered as a small definition list:

- **switch:** Type · Top housing · Bottom housing · Stem ·
  Spring (actuation g / bottom-out g) · Travel (mm) · Factory
  lubed · Released
- **keycap-set:** Profile · Material · Legend · Designer ·
  Released
- **board:** Layout · Case material · Mount style · Hotswap ·
  Wireless · Released

Released dates render via the existing `Intl.DateTimeFormat`
helper (medium format, UTC).

### `<PartBody>`

A single paragraph — the record's `description` field — set in
the article-body type scale. No MDX, no embedded components.
Just prose.

### `<MentionedInArticles>`

A rail of `<ArticleCard variant="row" />` instances for every
article whose `frontmatter.mentionedParts` references this
part. Heading: "Mentioned in N articles". Sorted by
publishedAt-desc. Empty-state: "Not yet mentioned in any
article. Check back as the catalog grows." (intentionally
honest).

### `<PartIndexCard>` (for the kind-index page)

Row layout: name (Mono) · brief one-liner from description
(truncated) · status pill. Anchor to `/part/<kind>/<slug>`.

## Data layer

A new helper in `@thock/content` (or
`apps/web/src/lib/data-runtime/`):

```ts
export function getArticlesMentioningPart(
  kind: 'switch' | 'keycap-set' | 'board',
  slug: string,
): Article[]
```

Returns articles whose `frontmatter.mentionedParts` array
contains `{ kind, slug }`. Sorted publishedAt-desc.

The `ResolvedPart` type already exists in `data-runtime` (lines
~40-60 of `index.ts`); reuse it for the page-level loader.

## SEO

- `generateMetadata({ params })` — title format
  `<Part Name> — thock`; description from the `description`
  field truncated to ~150 chars; canonical
  `/part/<kind>/<slug>`.
- `buildJsonLd(...)` with `@type: "Product"` for switches /
  keycap-sets (commercial-ish entities), `@type: "Thing"` for
  boards (less commerce-shaped). Include `name`,
  `description`, `manufacturer.name` (vendor), and
  `url`. Optional `releaseDate` from `releasedAt`.
- BreadcrumbList: thock → Part → `<kind>` → `<part-name>`.

The kind-index pages get `@type: "CollectionPage"` with
`hasPart` listing every part as a JSON-LD `Thing`.

## Decisions to lock at ship time

1. **Kind segment in the URL** — `/part/<kind>/<slug>` (chosen)
   vs `/part/<slug>` with kind discovery from data. Picked the
   explicit kind segment because (a) self-documenting URLs,
   (b) loader doesn't need a 3-way kind lookup on every
   request, (c) JSON-LD `@type` follows kind directly.

2. **Kind-index page** — ship it (chosen) vs defer to phase
   22+. Picked ship: it's <30 lines of code per kind index, it
   improves SEO faceting, and a missing index page makes the
   `/part/<kind>` URL feel half-built.

3. **OG images per part** — defer (chosen) vs ship now.
   Picked defer: not blocking the phase, and the next
   `/ship-asset` drain pass can backfill SVG art for the
   parts catalog as a separate concern.

4. **`MentionedInArticles` empty-state copy** — "Not yet
   mentioned in any article" (chosen) vs hide the section.
   Picked render: keeps the page's structure honest at a
   glance, and a reader who clicks through from search /
   sitemap knows the page exists for the part even when no
   articles cite it yet.
