# Phase 23 — Group-buy hero art

> Detailed brief for the autonomous loop. Generated by
> `/ship-a-phase` 2026-05-10 from the build plan one-liner +
> the `bearings.md` "Group-buy hero art" durable rule (locked
> 2026-05-10 via /oversight, in response to user observation
> that "all group bys on /group-buys need to show an image").

## Outcome

Every group buy in `data/group-buys/` has a colorful hero SVG
under `apps/web/public/group-buy-art/<vendor>-<slug>.svg` with
a sibling `<vendor>-<slug>.svg.json` provenance file. The data
schema gains a `heroImage: string?` field; records reference
their SVG via that field. `/group-buys` cards, home
`<GroupBuysWidget>` rows, and any `<TrackerRow>` group-buy
slot render the hero. New group-buy records added via
`/ship-data` bundle a `brander` invocation in the same
commit (per the bearings rule).

## Why

User observation via `/oversight` 2026-05-10T14:18Z: "all
group bys on /group-buys need to show an image" + "generate
hero art per group by when they are added ALWAYS just like we
do for other content (make this a rule in your shipping)".
Mirrors the article hero-art directive (locked 2026-05-09)
already in production for articles.

## Schema additive

```ts
// packages/data/src/schemas/group-buy.ts
heroImage: UrlSchema.or(z.string().regex(/^\/group-buy-art\//)).nullable(),
```

Or simpler — just nullable string with no regex constraint at
schema level (validation tested via cross-ref check):

```ts
heroImage: z.string().nullable(),
```

The field is **nullable** so partial backfill never blocks a
green build. The bearings render-fallback rule says: when
`heroImage` is null, render a coral-tinted placeholder block
(never a broken image).

## brander brief shape (per SVG)

Per the bearings hero-art rule: 1200×750 canvas, ~2px stroke,
single accent color splash + warm-bronze theme accent. Subject
mapped from `productKind`:

- **board** → top-down keyboard outline silhouette, optional
  layout callout based on the board's reported layout (e.g.,
  Alice-style angled bar for nyawice; standard 80% for
  paper80).
- **keycap-set** → 4-6 keycap profile silhouettes in cluster,
  with one focal cap painted in the kit's accent color.
- **switch** → cross-section / housing exploded view (none in
  current backfill set; pattern reserved for future).

Splash hue: pick from each kit's character. The user's example
(Prussian Alert article hero) used "deep Prussian red" as a
focal accent. Group-buy splashes follow the same intent —
warm bronze base, one splash that hints at the product:
- nyawice: cat-ear silicone in orange/pink/baby-blue/lilac →
  pick coral or dusty rose (avoid stock primary colors).
- gmk-cyl-greg-2: Greg memes / khaki tone → ochre.
- ishtar-r2: warm gold/red / mythic → muted terracotta.
- king-of-the-seas: ocean blue accent kit → muted teal (only
  exception to the avoid-blue guideline; subject demands it).
- sweet-nightmare: candy-themed dark accent → dusty rose.
- paper80: warm parchment-cream / Whatever Studio → ochre.

Each brander call returns: SVG path, provenance JSON path.
Main agent updates the data record's `heroImage` field with
the path.

Component file: `apps/web/src/components/group-buys/GroupBuyRow.tsx`

### `<GroupBuyCountdownRow>` on home `<GroupBuysWidget>`

Currently renders a compact row with just text. Add a small

or smaller on mobile depending on layout density).

Component file: `apps/web/src/components/home/GroupBuyCountdownRow.tsx`

### `<TrackerRow>` group-buy slots

If `/trends/tracker` ever cites a group buy as a mover, that
row would show a small thumbnail too. Today no tracker rows
cite group buys directly — defer to the future iteration that
adds them.

### Render fallback

When `heroImage` is null, render a coral-tinted placeholder
block (`bg-accent-mu/20` or similar) instead of an `<img>`.
This is the "never show broken image" guarantee — partial
backfill never produces 404 hero blocks.

## Decisions to lock at ship time

1. **Schema field type** — `string().nullable()` (chosen) vs
   stricter `regex(/^\/group-buy-art\//).nullable()`. Picked
   the simpler shape; the path validation isn't worth the
   schema complexity. data:validate doesn't currently
   enforce path-shape on imageUrl either.

2. **Position of hero in `<GroupBuyRow>`** — left column
   (chosen) vs banner above. Left matches the
   `<ArticleCard variant="row">` shape and keeps the row
   visually scannable at desktop.

3. **Mobile hero on `<GroupBuyCountdownRow>`** — hide on
   mobile (chosen) vs render small. Picked hide; the home
   widget is dense and the countdown text + name is the
   essential affordance.

4. **Backfill scope** — all 6 existing records (chosen) vs
   defer to /ship-asset drain. Picked ship-now: the user's
   complaint was that "all group bys need to show an image",
   so partial backfill doesn't satisfy the user signal.
