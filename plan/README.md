# plan

The build plan for thock. Read by every shipping skill
(`/ship-a-phase`, `/ship-data`, `/iterate`, `/march`,
`/plan-a-phase`) to keep the autonomous loop coherent.

## Layout

```
plan/
├── README.md                            # this file
├── bearings.md                          # standing context: stack, URL contract, defaults, sub-agents
├── AUDIT.md                             # latest /iterate audit (rewritten on each audit pass)
├── CRITIQUE.md                          # external-observer findings (append-only via /critique)
├── steps/
│   └── 01_build_plan.md                 # at-a-glance status block + per-phase scope
└── phases/
    ├── phase_1_bootstrap.md             # detailed phase 1 brief
    ├── phase_5_article.md               # canonical page-family template brief
    └── phase_<N>_<topic>.md             # generated by /plan-a-phase or inline by /ship-a-phase
```

## How the loop reads this folder

1. **Status check** — read the "Status (at-a-glance)" block at the
   top of `steps/01_build_plan.md`. The first `[ ]` row is the
   next phase.
2. **Brief** — load `phases/phase_<N>_<topic>.md`. If absent,
   `/ship-a-phase` calls `/plan-a-phase` inline (or the agent
   follows `skills/plan-a-phase.md` §5) to generate one + commit
   separately.
3. **Bearings** — `bearings.md` holds stack pins, URL contract,
   visual defaults, sub-agent registry, and standing decisions.
   Read once per session.

## Where design lands

The user emits design exports asynchronously into `design/<family>/`
at the project root (sibling to `plan/`). The loop does not wait
for design — if the export isn't there yet, the next phase that
touches that surface integrates it when it lands.

## How the audit works

`/iterate` writes findings to `plan/AUDIT.md` with this format:

```markdown
# Site audit — <ISO date>

## Top 5 findings (scored)

### [8.1] <one-line description>
- category: <content-gaps | data-gaps | seo | links | a11y | tests | perf>
- impact: <0-10>
- ease: <0-10>
- next: <action — invocation, sub-agent, or follow-up>
```

Entries are flipped `[ ]` → `[x]` (with commit hash) as they're
addressed.

## Manually ticking phases

If a phase ships outside the loop (rare), update
`steps/01_build_plan.md` by hand: flip `[ ]` to `[x]`, append the
commit hash, and add a line to the "Phase log" section. The loop
relies on these markers.
